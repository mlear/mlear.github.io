An injection of SQL

As fledgling programmers, we all need to be aware of what SQL injection is, and how to properly program in order to prevent it. Basically, the only reason SQL injection is possible is either because the programmer didn?t know any better, or the programmer was lazy/sloppy.

- The Problem -

SQl injection refers to ?injecting? SQL code in order to have unauthorized access to the database the code controls. This is done, and no I am not giving examples on how to hack as the information is relatively simple to find online as is, by passing carefully crafted strings as input into the code, such as a username or password. If the underlying program was not set up correctly and just calls a query on the input, it may result in a change to the original query and allow the crafty person to write whatever code he wanted to. All of the sudden your user table ids are available, all of them. Your credit card numbers, your passwords, your usernames? everything in that table is fair game. Hopefully this individual is kind and doesn?t include a DROP TABLE command as well. The problem here is with directly using the input of the user in the query, or concatenating. There is no check to see if the input is even valid, it just plugs in whatever is typed, and if certain characters are used, then you have issues to deal with.

- The Solution -

There are multiple ways to combat this. You could try and escape out the unwanted characters in the input, but that?s leads to other problems as you may get rid of something that you weren?t supposed to. You could try and blacklist certain words or characters, but this also runs into the same problems. It seems that the preferred method, and one that I admittedly have not had the pleasure to test fully, is to use Parameterized Queries. Huh? Yah. What that means is that when you set up your code, you use parameters for your values. After you get your input, you set them equal to the parameters. Now, when you run your query, the original code runs as intended and fills in the parameters when called upon, and not potentially adding in additional code like the concatenation was doing before. Simple fix! As long as we train ourselves from the very beginning to always code in this fashion, we will all avoid headaches and liability in the future.
